TVM ONNX Passthrough Project – Full Master Document

This document is a clean, chronological record of the project development process, written as a streamlined explanation of what was built, why it was built that way, and how each stage connects to the original assignment prompt. The goal was not just to produce code that works, but to move through the project in a logical sequence: first proving that a Python package can be installed, then proving that Python can call into C++, then reshaping that package into a small TVM-like runtime interface, and finally making that interface delegate execution to ONNX Runtime. This is the clearest way to show both technical understanding and a sensible development process.

The work was done in a Linux environment, specifically on a server that behaves like an Ubuntu/Linux development setup for command-line work. That operating system choice matters because the assignment explicitly asks for a Python module with a C/C++ component linked into Python as a shared object file, which on Linux is typically a .so file. Even though the exact server was an SMU Linux machine rather than a fresh local Ubuntu install, the workflow followed the same Linux-style packaging and build process you would use on Ubuntu: terminal commands, virtual environments, CMake, pip-based installation, and native extension compilation.

Where this should be done: on Ubuntu/Linux or a Linux server. The project relies on Linux-native build behavior for Python extensions, so this kind of workflow is best developed and tested in a Linux environment rather than on macOS or Windows first.

1. Assignment Goal

The assignment asked for a Python module that is installable via pip or Conda, starts as a simple module with one task, is then modified to include a C/C++ component, and eventually exposes a TVM-style interface implemented in C/C++ on Linux. The prompt also made it clear that the inference logic itself should not be reinvented. Instead, the implementation should be pass-through, meaning the custom module serves mainly as a wrapper or bridge while ONNX Runtime performs the real ONNX execution.

That requirement strongly suggests an incremental build strategy. Jumping straight into a final runtime wrapper would make debugging much harder and would hide whether packaging, native compilation, Python imports, or ONNX inference were responsible for any issues. Because of that, the project was built in four layers: basic Python packaging, C++ extension integration, TVM-style interface design, and ONNX Runtime pass-through. Each layer solved one problem at a time.

2. Environment Setup on Linux

The project began with environment setup. Since this was done on a Linux server, the cleanest approach was to create a dedicated project folder and then isolate the Python dependencies inside a virtual environment. This keeps the package dependencies local to the project and avoids conflicts with whatever Python installation the server already provides.

Commands used

mkdir -p ~/tvm_project
cd ~/tvm_project

python3 -m venv .venv
source .venv/bin/activate

python -m pip install --upgrade pip
python -m pip install pybind11 scikit-build-core onnxruntime numpy

These commands establish the development workspace. The virtual environment is important because it provides a predictable Python package environment for building and testing. The dependencies also each had a specific role. pybind11 was used to expose C++ code to Python, scikit-build-core was used as a modern build backend so that pip install . could drive the native build, onnxruntime was the execution backend required by the assignment, and numpy was used to feed tensors into the model during testing.

3. Stage One – Starting as a Simple Python Module

The first conceptual version of the project was deliberately simple. Before introducing compiled code, it helps to think of the package as an ordinary Python module with a very small function such as add(a, b). That stage is useful because it clarifies the package layout and the idea of a user-facing API before any native extension complexity is introduced. Even though the final working project moved quickly into the compiled version, the development logic still followed this progression: begin with the simplest meaningful package shape and then replace the implementation underneath it.

At this point, the key idea was that the package should eventually allow a clean import like from tvm_onnx_passthrough import .... That means the package structure needs to be correct from the beginning. The source code was therefore organized under a src/ layout, which is a common packaging pattern because it keeps importable package files separate from build configuration and C++ source files.

4. Stage Two – Converting the Simple Module into a C++ Extension

After the package structure was clear, the next task was to replace the toy Python functionality with a compiled C++ extension. This is the stage that satisfies the assignment requirement about linking a C/C++ component into Python as a shared object. Instead of building the final runtime immediately, the project first used a very small C++ function, again something like add(a, b), simply to prove that the Python package could load and execute native code.

This is an important step academically because it demonstrates that the student understands the boundary between Python and C++. Python is being used as the user-facing layer, but the actual implementation lives in compiled native code. If that boundary works for a tiny function, it can then be expanded into a class interface. If it does not work, there is no point in building higher-level runtime logic yet.

pyproject.toml

[build-system]
requires = ["scikit-build-core>=0.10", "pybind11"]
build-backend = "scikit_build_core.build"

[project]
name = "tvm-onnx-passthrough"
version = "0.1.0"

requires-python = ">=3.9"
dependencies = ["onnxruntime", "numpy"]

[tool.scikit-build]

This file defines the package as something pip can build and install. It also tells pip that building the project requires pybind11 and the scikit-build-core backend. In other words, this file is what turns the directory into an actual Python package project rather than a loose folder of scripts.

CMakeLists.txt

cmake_minimum_required(VERSION 3.18)
project(tvm_onnx_passthrough LANGUAGES CXX)

set(PYBIND11_FINDPYTHON OFF)
find_package(pybind11 CONFIG REQUIRED)

pybind11_add_module(_native cpp/module.cpp)
target_include_directories(_native PRIVATE /usr/local/es7/include/python3.9)

install(TARGETS _native DESTINATION tvm_onnx_passthrough)

This file controls native compilation. The most important line is pybind11_add_module(_native cpp/module.cpp), which tells CMake to compile the C++ file as a Python extension module named _native. The explicit include path was needed on this Linux server so that the compiler could find the Python headers correctly. That is a server-specific detail, but it is still useful to document because it shows how the build was adapted to the actual Linux environment.

First C++ implementation

#include <pybind11/pybind11.h>

namespace py = pybind11;

int add(int a, int b) {
    return a + b;
}

PYBIND11_MODULE(_native, m) {
    m.doc() = "Native extension";
    m.def("add", &add, "Add two integers");
}

This code is intentionally small. Its purpose is not to be the final deliverable, but to prove that Python can import and call code compiled from C++. In a staged project like this, that is a very meaningful milestone because it demonstrates that packaging, compilation, shared-object loading, and the Python-to-C++ bridge are all functioning correctly.

Python wrapper files

# src/tvm_onnx_passthrough/api.py
from ._native import add

# src/tvm_onnx_passthrough/__init__.py
from .api import add

__all__ = ["add"]

The wrapper files exist so that users do not need to import the internal native module directly. Instead, they interact with the clean package-level API. That is a small design choice, but it makes the package feel intentional and organized rather than exposing build internals.

Build and test commands

That output proved that the package was pip-installable and that the C++ extension was correctly linked into Python on Linux.

5. Stage Three – Replacing the Toy Function with a TVM-Like Interface

Once the C++ extension worked, the next step was to stop thinking of the project as a toy arithmetic package and instead reshape it into something that resembles a runtime interface. The assignment refers to implementing the TVM interface in C/C++, but in practice that does not mean rebuilding the whole TVM framework. It means creating a simplified runtime-style object that behaves like a model execution interface rather than a one-off math function.

That is why the add() function was replaced with a TVMModule class. A class is a better abstraction for runtime execution because it can store state such as the model path and later expose methods like run(inputs). This is more aligned with how actual model runtimes are organized, where an object represents a loaded or loadable model and methods on that object perform work.

Updated C++ code for the class stage

#include <pybind11/pybind11.h>
#include <string>

namespace py = pybind11;

class TVMModule {
public:
    explicit TVMModule(const std::string& model_path) : model_path_(model_path) {}

    std::string model_path() const {
        return model_path_;
    }

private:
    std::string model_path_;
};

PYBIND11_MODULE(_native, m) {
    m.doc() = "Native TVM-like pass-through extension";

    py::class_<TVMModule>(m, "TVMModule")
        .def(py::init<const std::string&>())
        .def("model_path", &TVMModule::model_path);
}

Updated Python API files

# src/tvm_onnx_passthrough/api.py
from ._native import TVMModule

# src/tvm_onnx_passthrough/__init__.py
from .api import TVMModule

__all__ = ["TVMModule"]

Reinstall and test

python -m pip install . --force-reinstall

python -c "from tvm_onnx_passthrough import TVMModule; m = TVMModule('model.onnx'); print(m.model_path())"

This stage demonstrated that the project had progressed from a simple compiled function to a minimal runtime-style interface. The class itself did not run inference yet, but that was intentional. Separating “interface creation” from “execution implementation” makes the development easier to explain and easier to debug.

6. Stage Four – Making the Implementation Pass-Through to ONNX Runtime

This is the stage where the project became a true solution to the assignment. The prompt explicitly says that Linux already supports ONNX execution through the onnxruntime module and that there is no need to reinvent that functionality. In other words, the custom C++ layer should not implement tensor kernels or parse ONNX files manually. It should act as a wrapper that forwards the work to ONNX Runtime.

That is what “pass-through” means in this context. The custom package defines the interface, but ONNX Runtime performs the actual inference. This is both technically sensible and academically appropriate because it demonstrates that the student knows when to reuse a mature runtime rather than attempting to rewrite it from scratch.

Final C++ implementation

#include <pybind11/pybind11.h>
#include <pybind11/stl.h>
#include <string>

namespace py = pybind11;

class TVMModule {
public:
    explicit TVMModule(const std::string& model_path) : model_path_(model_path) {}

    std::string model_path() const {
        return model_path_;
    }

    py::object run(py::dict inputs) const {
        py::module_ ort = py::module_::import("onnxruntime");
        py::object session = ort.attr("InferenceSession")(model_path_);
        py::list output_names;
        return session.attr("run")(output_names, inputs);
    }

private:
    std::string model_path_;
};

PYBIND11_MODULE(_native, m) {
    m.doc() = "Native TVM-like pass-through extension";

    py::class_<TVMModule>(m, "TVMModule")
        .def(py::init<const std::string&>())
        .def("model_path", &TVMModule::model_path)
        .def("run", &TVMModule::run);
}

The most important method here is run(). Instead of implementing inference directly in C++, it imports the Python onnxruntime module, creates an InferenceSession using the saved model path, and then calls session.run() with the supplied inputs. This is a neat demonstration of a hybrid design: the outer interface is C++, but the project still leverages the existing Python runtime stack underneath.

Rebuild and test that the method exists

python -m pip install . --force-reinstall

python -c "from tvm_onnx_passthrough import TVMModule; m = TVMModule('model.onnx'); print(hasattr(m, 'run'))"

7. Creating a Small ONNX Model for End-to-End Testing

To demonstrate that the final wrapper worked, a tiny ONNX model was created for testing. The simplest useful test is one where the model performs a very easy operation that can be checked by hand. The model chosen here computes y = x + 2. That makes the expected output obvious and removes ambiguity from the test. If the input is 3.0, then the correct output must be 5.0.

Using a tiny custom model is good practice in projects like this because it gives complete control over both the model and the expected result. It also proves that the runtime is not merely importing successfully but actually running inference and returning numerical output.

make_test_model.py

import onnx
from onnx import helper, TensorProto, OperatorSetIdProto

x = helper.make_tensor_value_info("x", TensorProto.FLOAT, [1])
y = helper.make_tensor_value_info("y", TensorProto.FLOAT, [1])

c_tensor = helper.make_tensor(
    name="c",
    data_type=TensorProto.FLOAT,
    dims=[1],
    vals=[2.0],
)

const_node = helper.make_node(
    "Constant",
    inputs=[],
    outputs=["c_out"],
    value=c_tensor,
)

add_node = helper.make_node(
    "Add",
    inputs=["x", "c_out"],
    outputs=["y"],
)

graph = helper.make_graph(
    [const_node, add_node],
    "AddTwoGraph",
    [x],
    [y],
)

opset = OperatorSetIdProto()
opset.version = 13

model = helper.make_model(
    graph,
    producer_name="test-model",
    opset_imports=[opset],
)

model.ir_version = 10

onnx.save(model, "test_model.onnx")
print("Saved test_model.onnx with IR version", model.ir_version)

The explicit ir_version = 10 line is there for compatibility with the ONNX Runtime version available in the environment. That ensures the model file can be loaded successfully by the installed runtime.

Create the test model

python -m pip install onnx
python make_test_model.py

Output

Saved test_model.onnx with IR version 10

8. Final End-to-End Inference Test

The last step was to use the completed package exactly the way an end user would. This is the most important proof of correctness because it exercises the whole stack at once: Python import, pip-installed package, C++ extension, runtime interface, ONNX Runtime session creation, model execution, and output conversion back to Python.

Final test code

python - <<'EOF'
import numpy as np
from tvm_onnx_passthrough import TVMModule

m = TVMModule("test_model.onnx")
out = m.run({"x": np.array([3.0], dtype=np.float32)})
print(out)
EOF

Final output

[array([5.], dtype=float32)]

This output is exactly what should happen. The model computes y = x + 2, and since the input tensor was [3.0], the resulting output tensor is [5.0]. The reason the printed result appears as [array([5.], dtype=float32)] instead of simply 5.0 is that ONNX Runtime returns a list of output arrays, and in this case there is one output array inside that list.

By the end of the process, the assignment requirements were satisfied in a clear and technically meaningful way. The project is pip-installable, includes a C++ extension linked into Python as a Linux shared object, exposes a minimal TVM-like runtime interface through a TVMModule class, and uses a pass-through design where ONNX Runtime handles the real model execution. The test model confirmed that the wrapper works end to end.

From an academic perspective, the project demonstrates several skills at once: Python packaging, Linux-based native builds, C++/Python interoperability, interface abstraction, and thoughtful reuse of an existing ML runtime. Just as importantly, it shows a disciplined development process. The solution was not built all at once. It was built in stages that each proved one concept before moving to the next. That structure makes the project easier to explain and stronger as a submission.

11. Short Submission Summary

This project implements a pip-installable Python package on Linux with a C++ backend exposed through pybind11. Development began with a simple package concept, then extended to a compiled native module, then evolved into a TVM-like runtime interface through a TVMModule class. The final run() method uses a pass-through design, delegating ONNX execution to ONNX Runtime rather than implementing inference manually. A small ONNX test model verified the end-to-end pipeline by returning the expected result [array([5.], dtype=float32)] for an input representing the expression y = x + 2.
